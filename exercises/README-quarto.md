# Quarto coding exercises

This documents the `exercise-NN-quarto/` folders. They render the R + Python coding
exercises to HTML and PDF from Quarto. The older `exercise-NN/` folders (LaTeX `.Rnw`)
are a separate pipeline documented in `README.md`.

Quarto versions currently exist for exercises 05, 06, 07, 08, 09, 10, 11, 13.

## The idea in one paragraph

Each exercise has one document skeleton (`<topic>.qmd`) and three notebooks. Prose
(problem statements and solution text) lives once, language-independent, in a
`*_texts.ipynb` notebook. Code lives twice, once per language, in `*_R.ipynb` and
`*_py.ipynb`. The build stitches them together: prose is substituted into the skeleton
by placeholder, code is pulled in by cell label, and the whole thing renders twice, once
with solutions shown and once without.

## Files in a chapter

```
exercise-09-quarto/
  mv_opt_4.qmd              # skeleton: INSERT_ placeholders, code embeds, R/Python tabs
  sol_mv_opt_4_texts.ipynb  # prose: one markdown cell per block, first line "label: <key>"
  sol_mv_opt_4_R.ipynb      # R solution: labeled code cells + INSERT_ prose cells
  sol_mv_opt_4_py.ipynb     # Python solution: same structure
  Makefile                  # include ../quarto.mk
  _quarto/latex-math.qmd    # per-chapter copy of the math macros (see "Math macros")
  inserted/                 # GENERATED by `make prep` (do not hand-edit)
  logs/                     # GENERATED: replace_texts.py logs
  mv_opt_4_all.{html,pdf}   # GENERATED: solution version (all = with solutions)
  mv_opt_4_ex.{html,pdf}    # GENERATED: exercise-only version
```

Shared, at the `exercises/` level:

```
quarto.mk                   # per-chapter build rules (each chapter Makefile includes it)
scripts/replace_texts.py    # the prose-substitution step
_quarto.yml                 # Quarto project config (theme, html/pdf formats, mathjax)
_quarto/latex-math.qmd      # MASTER math macros; synced into each chapter with `make sync-math`
_quarto/colab-math.qmd      # reference macro set for the standalone-notebook math cell
```

## How the pieces connect

1. **Prose substitution.** Every markdown cell in `*_texts.ipynb` starts with a line
   `label: <key>`. `replace_texts.py` builds a `{key: text}` map from those cells and
   replaces every `INSERT_<key>` token in the `.qmd` and in the two code notebooks. This
   is strict on purpose: an `INSERT_<key>` with no matching label is a fatal error, so a
   typo fails the build loudly instead of shipping a literal `INSERT_...` string. An
   unused label only warns.

2. **Code embedding.** Code cells carry a `#| label: <name>` directive. The skeleton
   pulls a cell in with `{{< embed sol_mv_opt_4_R.ipynb#<name> echo=true >}}`. Only the
   labeled code cells are embedded; the notebooks' own prose cells are not used by the
   `.qmd` render (they exist so the notebooks also stand alone).

3. **Solution gating.** Solution blocks are wrapped in
   `::: {.content-visible when-profile="solution"} ... :::`. The solution render passes
   `--profile=solution` and shows them; the exercise-only render omits them.

4. **R / Python tabs.** Language pairs use
   `::: {.panel-tabset} ### R <embed> ### Python <embed> :::`.

## Build

Per chapter (run from inside `exercise-NN-quarto/`):

```bash
make execute   # run the source notebooks in place to cache cell outputs (plots, prints)
make prep      # run replace_texts.py -> writes substituted files into inserted/
make solhtml   # render inserted/<topic>.qmd with --profile=solution -> <topic>_all.html
make exhtml    # render without the profile                          -> <topic>_ex.html
make solpdf    # solution PDF -> <topic>_all.pdf
make expdf     # exercise PDF -> <topic>_ex.pdf
make all       # execute + prep + all four renders
make copy      # copy the html/pdf up to ../../exercises-pdf/
make clean     # remove inserted/ and Quarto build artifacts
```

Across all chapters (run from `exercises/`):

```bash
make quarto-all              # render every quarto chapter
make quarto-changes          # render only chapters with changes vs origin/main
make init-quarto-chapter NN=06   # scaffold a new exercise-06-quarto/ (Makefile + _quarto/)
make sync-math               # re-copy _quarto/latex-math.qmd into every chapter
```

### When do I need `execute`?

`execute` re-runs the notebooks and re-caches their outputs. It is the slow step and it
needs the kernels (Python, plus IRkernel registered as `ir` for R).

- Changed only prose or a code comment -> `make prep` and a render. No `execute`.
- Changed code that changes an output -> `make execute`, then `make prep` and a render.

Gotcha: clearing a code cell's `outputs` commits you to re-execute. If you clear outputs
and then only `prep` + render, that cell's plot or printout will be gone from the HTML.
So for comment-only edits, do not clear outputs.

## Math macros

There are two independent math-macro sources for two independent render paths:

- **The `.qmd` render** uses `{{< include ../_quarto/latex-math.qmd >}}`. The render runs
  on `inserted/<topic>.qmd`, so `../_quarto/` resolves to the chapter's own
  `_quarto/latex-math.qmd`. That is why each chapter keeps a copy. Do not edit the copy
  directly: edit `exercises/_quarto/latex-math.qmd` (the master) and run `make sync-math`.

- **A notebook opened on its own** (VS Code, Jupyter, Colab) renders its first markdown
  cell, which defines the macros it uses. Those definitions use `\providecommand`, which
  is idempotent, so re-rendering the cell does not throw a KaTeX "attempting to redefine"
  error. Use `\providecommand`, not `\newcommand`, in that cell.

## Reproducibility

Each `.qmd` opens with a collapsed "Reproducibility" callout listing the pinned package
versions (pip for Python, `remotes::install_version` for R) used to produce the outputs.

## Things that are generated

- `inserted/` and `logs/` in each chapter: rebuilt by `make prep`.
- The `_all`/`_ex` `.html` and `.pdf`: rebuilt by the render targets (the copies under
  `exercises-pdf/` are the committed artifacts, via `make copy`).
- `_quarto/latex-math.qmd` inside a chapter: a copy, kept in sync from the master.
